1. The problem

Quasi-sparsity: a few channels dominate the scale

The vector is not sparse because most entries are nonzero. But it is quasi-sparse because two channels carry almost all of the magnitude. A normal low-bit quantizer must cover the whole range from about -10 to +10, so subtle values around 0.1 get crushed into coarse buckets.

The running vector has min = -10.20 and max = 9.80, so 16 bins must cover a 20-point range. The step is (9.80 - -10.20) / 15 = 1.333.

2. Random rotation

Same geometry, less spiky coordinates

An orthogonal random rotation changes the coordinate system, not the information. Distances, angles, inner products, and cosine relationships are preserved when compatible vectors are rotated the same way. What changes is that the energy gets spread across many coordinates instead of sitting in a few outlier channels.

3. PolarQuant

Store each pair as magnitude plus direction

PolarQuant groups coordinates into pairs and rewrites each pair as a radius and an angle. A pair like (0.15, 9.80) no longer means "one tiny value beside one huge outlier." It means "a long vector pointing almost vertical."

The radius carries magnitude. The angle carries how that magnitude is split between the two coordinates. The small coordinate survives because it changes the angle.

4. QJL residual bits

The leftover error becomes signs

PolarQuant captures most of the vector. It still leaves a small residual error. TurboQuant's product-oriented version uses a one-bit Quantized Johnson-Lindenstrauss correction: project the residual in random directions and store only whether each projection is positive or negative.

The residual is just the quantization mistake left after dequantizing the PolarQuant code: residual = original vector - dequantized PolarQuant vector. For the running example, if d4 starts as 9.800 and PolarQuant reconstructs it as 9.807, then the residual at that dimension is 9.800 - 9.807 = -0.007.

QJL is not trying to store every residual coordinate exactly. It takes random dot products with that residual and stores only the sign of each result. Those signs preserve useful correction information for inner products without storing another full vector.

Each bit is 1 when one random projection of the residual is positive, and 0 when it is negative. During scoring, those signs help correct the bias left by the quantized approximation.

KPI headers

Self R@1 / Self R@10
Self-query recall against exact FP32 top-10. R@1 means the original vector is ranked first. R@10 is mean top-10 overlap; a 9/10 overlap is 90%, not 100%.
Random R@10
Mean fraction of the 10 exact FP32 top-10 neighbors recovered by the approximate top-10 for deterministic normalized random mixture queries. Random R@1 is intentionally omitted.
ms/query
Milliseconds per query, not microseconds. Query-major A completes one independent query across every bounded chunk/range or graph candidate frontier before the next. Batched B divides a reused 2,000-query workload by 2,000 and is throughput only.
Data store
All payloads are disk-backed; the complete 50K/100K stores are not resident in RAM. HNSW adjacency is compact and resident, while its payload vectors remain on disk.
Vector staging
Method-specific vector payload held for the active disk chunk/range. FP32 uses raw f32, FP16 uses raw half bits with conversion during scoring, and low-bit rows use their compressed active ranges.
Search RAM
Active vector payload in RAM only, not overall app RAM. UI, runtime/dependency libraries, allocator overhead, OS page cache, persisted ROM, and temporary build peaks are outside this KPI.

Raw FP32 chunk vectors

What each method does

FP32 exact
Brute-force exact dot-product scan over all vectors in the selected dataset. It is used as ground truth for recall.
FlatIndex methods
FlatIndex scans every vector; HNSW traverses a compact M=16 graph. Both families have A query-major and B batched-throughput paths, with the same 30/50 MiB pure-vector caps.
HNSW parameters
M=16, efConstruction=128, efSearch=1024, max layers 8, base degree at most 32. The larger search frontier improves navigation recall, but HNSW 8/4 still show same-bit quantization loss.
FP16 disk scan
Persists IEEE FP16, reads one bounded half-bit chunk, converts four values at a time with ARM FP16/NEON instructions during the dot product, then releases the chunk.
8-bit TurboQuant
Uses rotation, TQ+ calibration, byte-code packed storage, and an ARM block-major byte scorer. Its main byte-indexed centroid lookup loop is scalar, so it can be slower than FP16 despite smaller ROM; query-major ARM also uses bounded coarse-to-exact same-bit scoring.
4-bit TurboQuant
Uses bit-plane packed codes, blocked 32-vector layout, per-query nibble LUTs, and ARM NEON lookup accumulation. The kernel is efficient, but this one-query KPI also includes loading and preparing every bounded range.
GPU
No GPU number is claimed: the APK has no verified Vulkan/NNAPI search kernel, and this disk/candidate-I/O workload would make unmeasured dispatch and transfer overhead misleading.

How the benchmark actually runs

Four paths, four different scheduling contracts

Both index families use the same 30/50 MiB pure-vector cap, but A and B intentionally answer different performance questions. Quantized paths also obey a strict same-bit rule: their search and final top-10 come from the measured representation alone.

A1

FlatIndex A: one real request

For one query, read every bounded raw chunk or compressed range, score all vectors in that range, merge top-10, release it, and continue. Only after the query completes does the next request begin.

B1

FlatIndex B: range-major batch

Load one range once, scan it against all 2,000 queries with eight Rayon workers, update one heap per query, release the range, and move on. This is why B has excellent amortized throughput.

A2

HNSW A: graph then payload

Traverse the resident M=16 graph with efSearch=1024, read candidate payload spans from disk, score the selected bit-width, finish top-10, and then start the next query.

B2

HNSW B: shared graph batch

Eight workers share the graph and reuse bounded candidate caches while processing 2,000 queries. The reported ms/query is batch time divided by 2,000, not isolated request latency.

Each FlatIndex bit flow

FP32
Read bounded raw .f32 chunks and compute exact 768-dimensional dots.
FP16
Read persisted half bits and convert four values at a time through the arm64 NEON dot loop; no full FP32 database is resident.
8-bit
Rotate/calibrate the query, build byte LUTs, scan block-major codes, and finalize with same-bit 8-bit scores. ARM query-major uses a bounded coarse candidate pass.
4-bit
Rotate/calibrate the query, build nibble LUTs, scan two nibbles per byte in 32-vector blocks with NEON, and finalize with same-bit 4-bit scores only.

Each HNSW payload flow

Shared graph
Build once over persisted FP16 navigation vectors; adjacency stays resident while payload stores stay on disk.
FP32 / FP16
Use graph-selected IDs, then score candidates from the matching disk-backed FP32 or FP16 payload.
8-bit / 4-bit
Use graph-selected IDs, read the matching compressed payload, and finalize from same-bit scores. No raw-FP32 rescue pass is allowed.
Why recall differs
efSearch can improve candidate coverage, but it cannot restore precision lost by the compressed payload; sparse candidate reads also raise latency.

Why native Rust is fast

Low latency comes from memory layout and NEON-friendly kernels

1

Rust JNI, not Java loops

A foreground service enters native Rust once. Dot products, flat top-k scans, quantization, heaps, blocked layout, and search kernels run in optimized release Rust.

2

Blocked 32-vector layout

prepare() repacks compact codes into 32-vector blocks so a query scans contiguous bytes instead of chasing one vector at a time. The active block remains bounded by the pure-vector cap.

3

NEON nibble LUT path

4-bit search uses ARM NEON table lookups and vector accumulators. The query stays FP32, but database codes remain compressed and final ranking stays 4-bit. The LUT kernel is fast; disk range staging and quantization error still affect the end-to-end result.

4

Why 8-bit can be slower

On arm64, FlatIndex A FP16 uses native FP16-to-FP32 conversion plus NEON dot products. The 8-bit scorer still performs byte-indexed centroid lookups for each dimension, so it can lose to FP16 despite smaller ROM. 4-bit has a dedicated NEON nibble-LUT kernel, but its query-major result still includes per-range staging and its lower precision is visible in recall.

Application 2

TurboQuant for the LLM KV cache

During generation, every previous token leaves key and value vectors in the KV cache. Long context makes that cache huge. Quantizing it reduces memory bandwidth and lets the model keep longer histories or serve more users.

The catch is accuracy: attention depends on dot products between the current query and cached keys. TurboQuant targets this by preserving geometry and correcting residual inner-product bias.

Autoregressive memory grows with context

KV-cache mechanics

What happens at store time and attention time

TurboQuant does not replace attention. It changes how past keys and values are stored. New K/V vectors are compressed as they enter the cache. Later, attention kernels read compressed blocks, reconstruct approximate K/V values only for the current computation, and discard those temporary values.

store time

1. Compute the new token's K and V

token t K_t fp16/bf16 V_t fp16/bf16

The model creates normal full-precision key and value vectors for the newest token.

compress

2. Rotate, PolarQuant, then QJL

K_t R polar codes 01011001

Random rotation smooths channels, PolarQuant stores radius plus angle, and QJL stores the residual as sign bits.

cache memory

3. Store compressed K/V codes

K cachecodes + bits
V cachecodes + bits

The long-context memory footprint shrinks because the cache keeps compact codes instead of full tensors.

query time

The current query stays high precision

The query is temporary, so there is little memory benefit in storing it compressed. If cached keys live in rotated space, the query is rotated at runtime before scoring.

q_rot = R q_current
compressed K blocks load on-the-fly dequant score q_rot dot K_hat
compressed V blocks load on-the-fly dequant mix sum weights * V_hat

Is dequantization needed?

Yes, but usually it is fused into the attention kernel. The full cache is not permanently expanded. A block is loaded, dequantized into registers or fast local memory, used for dot products or value accumulation, then discarded.

Keys and values have different jobs

Keys must preserve attention scores, so the rotated query and reconstructed keys must keep dot products close. Values must preserve the final weighted representation, so the value approximation must be accurate after softmax weights are applied.

Why this helps long context

Long prompts repeatedly read old K/V vectors. Compression reduces cache memory and memory bandwidth, which can unlock larger context windows, higher batch sizes, or faster attention on suitable kernels.

cache stores codes; attention uses K_hat and V_hat temporarily The model still computes normal attention, but the cache representation is compact.

Reported results

What public reports claim

The TurboQuant paper reports quality neutrality around 3.5 bits per channel for KV-cache quantization and marginal degradation around 2.5 bits per channel. Public Google coverage reported long-context evaluations such as Needle in a Haystack, LongBench, RULER, ZeroSCROLLS, and L-Eval on open models including Gemma and Mistral. Media coverage also reported at least 6x KV-cache memory reduction and up to 8x faster attention-logit computation on H100 GPUs.
